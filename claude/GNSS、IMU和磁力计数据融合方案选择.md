stm32端连接了GNSS, IMU,磁力计，然后stm32端和树莓派zero 2w 通过串口相互连接，现在想做GNSS,IMU和磁力计的数据融合，这个数据融合是在stm32端做好，还是在树莓派zero 2w上做好呢，数据融合可以使用什么方法来做呢

**Claude**

这是个典型的组合导航系统架构问题。先说结论，再展开。

### 放哪里做：取决于你的 STM32 型号和融合层级

关键的硬件前提是 **你的 STM32 有没有 FPU（浮点运算单元）** 。

- F1/F0（Cortex-M0/M3）没有硬件 FPU，跑卡尔曼滤波会很吃力，矩阵运算靠软件模拟浮点会很慢。
- F4/F7/H7（Cortex-M4F/M7）有单精度（甚至双精度）FPU，跑 ESKF、Madgwick 这类算法完全没问题，几百 Hz 轻松。

而树莓派 Zero 2W 是四核 Cortex-A53 @1GHz、512MB 内存，跑 Linux，算力远超任何 STM32，但 **它不是实时系统** ——Linux 调度有抖动，拿不到精确的 IMU 采样时间戳。

### 我的建议：核心融合放 STM32

对 GNSS/IMU/磁力计这种组合导航， **时序确定性比算力更重要** ，所以倾向于在 STM32 上做核心融合，理由：

1. **时间戳精确** 。IMU 高速积分（陀螺、加速度计积分推算姿态/位置）对采样间隔的精度极其敏感。STM32 能用硬件定时器/中断给每个 IMU 样本打准确时间戳，而数据先经串口传到树莓派再处理，时序会被 UART 缓冲和 Linux 调度抖动污染，积分误差会累积。
2. **高采样率无 OS 抖动** 。IMU 通常要 100–500Hz 甚至更高，STM32 能稳定跑，树莓派上的用户态进程做不到稳定的硬实时。
3. **避免串口瓶颈** 。如果在树莓派融合，你得把原始 IMU 数据以高速率串口流过去。比如 200Hz、9 轴浮点数据，加帧头校验，约需 80–100kbps，115200 波特率已经比较紧张，建议 460800 或更高。而如果 STM32 做完融合只把结果（位姿，几十 Hz）发过去，串口压力小很多。

**典型的分工（推荐）** ：

```
STM32（实时层）          串口          树莓派（应用层）
 ├ 读 GNSS/IMU/磁力计  ──────────→  ├ 接收融合后的位姿
 ├ 时间戳同步                       ├ 路径规划/建图/记录
 ├ ESKF 核心融合                    ├ 调试/可视化/网络
 └ 输出位姿(20-50Hz)                └ 高层算法
```

什么时候反过来、放树莓派做？如果你后续要做 **视觉惯性融合（VIO）、SLAM、因子图优化** 这类重计算，或者你处于快速原型阶段想用 Python 方便调试和记录数据，那就把原始数据传到树莓派融合。开发迭代确实比在单片机上方便得多。

### 用什么融合方法

按融合目标分两层来选：

**只要姿态（roll/pitch/yaw，AHRS）** ，用陀螺 + 加速度计 + 磁力计：

- **Mahony / Madgwick 互补滤波** ——最轻量，几行核心代码，M4 上几百 Hz 没压力，是最常见的入门选择。Madgwick 收敛快，Mahony 可调性好。
- 想要更高精度可以用基于 EKF 的 AHRS（状态里带陀螺偏置）。

**要完整导航解（位置 + 速度 + 姿态）** ，融合 GNSS：

- **误差状态卡尔曼滤波（ESKF, Error-State Kalman Filter）** ——这是组合导航的事实标准，强烈推荐。结构是：IMU 做高频预测（捷联惯导机械编排），GNSS 来位置/速度观测做修正，磁力计提供航向修正。典型 15 状态（位置、速度、姿态、加计偏置、陀螺偏置）。比直接 EKF 数值更稳定，尤其姿态用误差量表示能避免奇异性。
- **松耦合 vs 紧耦合** ：松耦合用 GNSS 模块解算好的位置/速度做观测，实现简单，绝大多数场景够用；紧耦合用原始伪距/载波，在卫星信号受遮挡时更鲁棒，但复杂得多，且需要 GNSS 模块能输出原始观测量。 **先做松耦合** 。
- UKF 也可以，但对这个问题相对 ESKF 没明显收益，计算量更大，一般不必要。

### 几个容易踩的坑

- **时间同步** ：无论放哪边做，GNSS（1–10Hz）和 IMU（高频）的时间对齐是精度的命门。GNSS 的 PPS（秒脉冲）信号最好接到 STM32，用来校准时钟。
- **磁力计标定** ：磁力计必须做硬磁/软磁标定（椭球拟合），否则航向会有几十度误差。装在载体上还要避开电机、电流大的线缆。
- **坐标系与杆臂** ：三个传感器的安装位置、坐标系定义要统一，GNSS 天线相对 IMU 的杆臂（lever arm）在高动态下不能忽略。

### 厘米级的前提：你的 GNSS 必须是 RTK（或 PPP）

普通 GNSS 模块（NEO-M8N 这类单频模块）的绝对定位精度是 2–5 米， **无论你怎么融合 IMU 和磁力计，都不可能把它变成厘米级** 。原因是 IMU 只能做航位推算（dead reckoning），它在 GNSS 更新之间做平滑、在短暂失锁时维持位置，但它本身会快速漂移——纯惯导维持厘米级精度顶多撑几秒。所以融合改善的是 **输出频率、姿态、和短时连续性** ，绝对精度的天花板完全由 GNSS 决定。

要厘米级，你的 GNSS 子系统必须是：

- **RTK（实时动态）** ：双频模块（如 u-blox ZED-F9P）+ 基准站差分改正数据（通过 NTRIP 网络 RTK，或自建基站）。这是达到 cm 级最现实的路线。
- 或 PPP，但收敛慢、需要精密星历改正，工程上更麻烦。

**所以第一个问题：你现在的 GNSS 是什么模块？是否是 RTK，有没有改正数据源？** 如果不是，融合做得再好也到不了 cm 级，得先换硬件。

### 假设你已经有 RTK，再谈融合放哪里

STM32F405RG 是 Cortex-M4F，168MHz，1MB Flash，192KB RAM， **带 FPU** ，跑 ESKF 算力上没问题。我上一条说的"实时层放 STM32"的理由（时间戳精确、无 OS 抖动）在 cm 级要求下其实更成立——cm 级对时间同步极其敏感。

但 cm 级有一个 **专门针对 F405 的大坑必须提醒你** ：

> F405 的 FPU 只支持 **单精度（float32）** 。float32 尾数 24 位，约 7 位十进制有效数字。如果你用 ECEF 坐标表示位置，地球半径约 6,378,137 米，光是米级就占满 7 位， **float32 在这个量级下的分辨率约 0.5 米，根本表示不了厘米** 。

解决办法：

- 导航滤波器内部用 **局部坐标系（ENU/NED）** ，原点设在作业区域附近，坐标值保持在几公里以内，这样 float32 的 cm 精度才够用（距原点 1km 时 float32 分辨率约 0.1mm，没问题；但作业范围大到几十上百公里就不行了）。
- 或者位置相关的核心计算用 **双精度（double）** ——F405 的 FPU 不支持双精度，会软件模拟，慢一些，但 168MHz 在中等更新率下扛得住。

这个坑很多人在单片机上做 cm 级导航时栽过，务必注意。

### 融合方法：松耦合 ESKF

对 RTK + INS，标准做法是 **误差状态卡尔曼滤波（ESKF）的松耦合** ：

- IMU 高频做捷联惯导预测（机械编排）；
- RTK 输出的位置/速度作为观测做修正；
- 磁力计提供航向修正。
- 典型 15 状态：位置、速度、姿态、加计偏置、陀螺偏置。

先做松耦合（用 RTK 解算好的位置/速度），实现相对简单，cm 级场景绝大多数够用。紧耦合（用原始伪距/载波）在遮挡下更鲁棒但复杂得多，且需要 GNSS 输出原始观测量——先别碰。

### 一个航向的坑

单天线 RTK + IMU，载体 **静止或低速时拿不到可靠航向** ，得靠磁力计（而磁力计必须做椭球标定，且要远离电机/大电流，否则航向几十度误差）。如果对航向要求也高，可以考虑 **双天线 moving-baseline RTK** （如双 F9P），直接给出 cm 级基线的航向，比磁力计可靠得多。

### 放哪里做：核心融合放 F405，结论很明确

F405 跑 15 状态 ESKF 毫无压力：每个 15×15 矩阵才 900 字节，192KB RAM 绰绰有余；168MHz M4F 单精度运算 200Hz 预测 + 10Hz 更新轻松。

务实的做法是： **先在树莓派上用 Python 把 ESKF 逻辑调通** （方便记录原始数据、反复回放调参、画图看残差），逻辑对了再移植回 F405。最终部署一定在 F405。

### 算法：松耦合 ESKF（误差状态卡尔曼）

在 **本地导航系** （NED 或 ENU，原点设在作业区附近，务必避开 ECEF 量级的 float32 精度坑）里做。

**状态分两套。** 标称状态（nominal）用高精度积分，误差状态（error-state）才是 KF 真正估计的量：

```
标称状态 x:  p(位置3) v(速度3) q(姿态四元数4) b_a(加计零偏3) b_g(陀螺零偏3)
误差状态 δx: δp(3)    δv(3)    δθ(3,姿态误差) δb_a(3)        δb_g(3)   → 共15维
```

用误差状态的好处：姿态用 3 维 δθ 表示，避免四元数的过参数化和奇异性，数值更稳。

**预测（IMU 速率，比如 200Hz）——捷联惯导机械编排：**

```
姿态:  q ← q ⊗ Exp((ω_m − b_g)·Δt)          // ω_m 陀螺原始值
速度:  v ← v + (R(q)(a_m − b_a) + g)·Δt      // a_m 加计原始值, g 当地重力
位置:  p ← p + v·Δt
```

误差状态本身不积分（保持 0，等观测来修正），但协方差要传播：

```
P ← F·P·Fᵀ + G·Q·Gᵀ
```

F 是误差状态的状态转移矩阵，关键的非对角项：δv 受 δθ 影响（项为 −R·\[a\_m−b\_a\]×·Δt）、受 δb\_a 影响（−R·Δt）；δθ 受 δb\_g 影响（−R·Δt）；δp 受 δv 影响（I·Δt）。Q 来自 IMU 噪声：加计/陀螺的白噪声 + 零偏的随机游走， **这四个数直接决定融合好坏，要按你 IMU 的 datasheet（noise density、bias instability）填，再微调** 。

**RTK 位置/速度更新（GNSS 速率，1–10Hz）：**

```
残差:  r = p_rtk − (p_ins + R(q)·l_arm)       // l_arm 是天线相对IMU的杆臂
       (有速度观测就再加 v_rtk − v_ins)
H:     位置对 δp 的雅可比为 I；含杆臂时对 δθ 有 −R·[l_arm]× 项
K = P·Hᵀ·(H·P·Hᵀ + R_meas)⁻¹
δx = K·r
P ← (I − K·H)·P
```

**误差注入 + 重置** （ESKF 的精髓）：把估计出的 δx 注入标称状态，然后清零 δx——

```
p ← p + δp;  v ← v + δv;  q ← q ⊗ Exp(δθ);  b_a ← b_a + δb_a;  b_g ← b_g + δb_g
δx ← 0;  P ← Gₓ·P·Gₓᵀ   // Gₓ 处理姿态误差重置, 近似可取 I
```

**磁力计更新** ：只修正航向（yaw），不要让磁场直接改 roll/pitch（会被磁干扰污染）。即用磁力计算出的航向角和 INS 航向做残差，H 只映射到 δθ 的偏航分量。 **前提是磁力计先做椭球标定，且远离电机/大电流。**

### 几个对你这套系统最关键的工程点

**用 RTK 状态标志动态调 R\_meas，这是星基 RTK 融合的命门。** 模块会输出解状态（fixed / float / DGPS / single）和精度估计。fixed 时 R 给很小（cm 级，比如 0.02m），float 时给 dm~m 级，掉到 single 时 R 给到几米甚至直接 **不更新只靠纯惯导** 。如果掉到 float 还按 cm 级信任它，meter 级的"伪 cm"观测会瞬间污染整个滤波器。务必做这个门控。

**PPS 时间同步** ：把 GNSS 的秒脉冲接到 F405 的定时器输入捕获，用来对齐 IMU 时间戳和 GNSS 历元。星基 RTK 的解算有传输延迟，残差更新时要补偿这个时延（用对应历元时刻的 INS 状态做残差，不是当前时刻）。

**float32 局部坐标系** ：再强调一次，位置一定在以作业区为原点的本地系里算，坐标值控在几公里内，cm 级才表示得了。作业范围若上百公里，得分块换原点或位置项用 double 软件浮点。

**初始对准** ：静止时用加计求 roll/pitch，磁力计求初始 yaw；先让星基 RTK 收敛到 fixed 再开始信任位置观测（星基重收敛可能要几十秒到几分钟，看你的服务和环境）。

**低速/静止用 ZUPT** ：单天线 + 静止时 RTK 给不出航向、IML 零偏也难观测。检测到静止时加零速观测（v=0），能有效约束零偏漂移。如果航向精度要求也高，强烈建议上 **双天线 RTK** 直接出 cm 级基线航向，比磁力计可靠得多。

### 我倾向 F405 的理由（这些是真实存在的）

**时间戳的确定性** 。这是最硬的理由。ESKF 的 IMU 预测靠的是 `位置 += 速度 × Δt` 、 `姿态 ⊗ Exp(ω·Δt)` 这种积分，Δt 的误差会直接进入状态。F405 能用硬件定时器给每个 IMU 样本打微秒级时间戳。而数据要先走 SPI/I2C 采集、再经 UART 传到树莓派，这个链路上 UART 缓冲、Linux 内核调度都会引入抖动——树莓派跑的是普通 Linux，进程随时可能被调度器换出几毫秒。200Hz 时标称 Δt 是 5ms，几 ms 的抖动就是相对误差很大的污染。

不过要诚实：这个问题 **可以缓解** 。如果让 F405 在采样时就打好时间戳、连同数据一起发给树莓派（树莓派用 STM32 给的时间戳而不是自己的接收时刻），抖动问题就基本绕过了。所以这条理由更准确的说法是"在树莓派做需要你正确处理时间戳传递"，而不是"做不了"。

**RTK 失锁时的惯导桥接** 。星基 RTK 掉到 float/single 时全靠 INS 维持 cm 级，这段纯惯导推算最怕时序抖动。理由本质同上。

**串口带宽** 。在树莓派融合，原始 IMU 得高速率流过去；在 F405 融合则只发结果（几十 Hz 位姿）。但这也不是大问题——把波特率提到 460800 或更高就行。

**系统简洁性 / 上电即用** 。F405 方案是个自包含的实时导航单元，没有 OS 启动、文件系统损坏、SD 卡掉电写坏的风险。对要长期可靠运行的部署有意义。

### 但树莓派方案有它实打实的优势

我前面把这些讲得太轻了，实际上对很多人它们更重要：

**开发效率天差地别** 。Python + NumPy 写 ESKF，矩阵运算就是 `P @ H.T` ，调试时随便 print、画残差曲线、Jupyter 里交互。F405 上你要手写矩阵库或用 CMSIS-DSP，调试靠串口打印或 SWD,改一版要重新编译烧录。原型阶段树莓派可能快 10 倍。

**算力和精度上限更高** 。Zero 2W 是四核 A53,跑 double 双精度原生支持——直接绕过我反复强调的 **F405 float32 表示不了 ECEF 厘米** 那个坑。你可以用全局坐标、更高维状态、更复杂的模型。

**可扩展性** 。如果未来要加视觉、做 VIO/SLAM、因子图优化、数据记录回放,这些只能在树莓派做。在树莓派融合,这些是自然延伸；在 F405 融合,等于把核心逻辑放在了扩展不了的地方。

**数据记录与复盘** 。树莓派能把所有原始数据存盘,出了问题可以离线回放调参。这对调一个 cm 级系统极其有价值。

### 所以真正的判断标准

不是"哪个更好",而是:

- **追求确定性、要做成可靠的嵌入式产品、动态较高、不需要扩展重算法** → F405,接受开发慢一点。
- **还在研究/原型阶段、要快速迭代、未来要加视觉或重算法、想用 double 避开精度坑** → 树莓派,但你必须正确处理 IMU 时间戳(让 STM32 打戳),并接受非实时带来的抖动需要工程上去管理。

还有第三条路,其实是很多成熟系统的做法—— **分层**:F405 只做高频 AHRS(姿态,Mahony/Madgwick,对时序最敏感的部分)和精确打戳,把带时间戳的数据上传;树莓派做位置层的 GNSS/INS 松耦合融合。这样时序敏感的部分留在实时端,算力和灵活性需求高的部分放 Linux 端,两边的长处都拿到了。

**最关键的一点先说**:让 STM32 给每个样本打时间戳,数据帧里带上这个 ts。树莓派 **绝不能用"我收到这帧的时刻"当采样时间** ——UART 缓冲 + Linux 调度会让接收时刻抖动好几毫秒,而 ESKF 的 IMU 积分 `Δt` 直接决定精度。用 STM32 的硬件时间戳算 Δt,就把 Linux 非实时的影响基本绕过去了。这是树莓派方案能不能成立的命门。

### 第一步:让接收尽可能稳

虽然 Linux 不是硬实时,但能做几件事让它"够实时":

- 串口接收单独开一个线程(或进程),只干一件事:读字节、找帧头、校验、解析、塞进队列。不要在这个线程里做融合计算。
- 给这个进程提优先级:`chrt -f 80 python3 fusion.py` (SCHED\_FIFO),或代码里 `os.sched_setscheduler` 。再配合 CPU 亲和性把它绑到一个核,`taskset -c 3` 。
- 关掉串口的 Python 层缓冲延迟,`pyserial` 设 `timeout` 小一点,或用 `select` 。
- Zero 2W 四核,可以把"采集线程 / 融合线程 / 记录线程"分开,GIL 影响可以用多进程(`multiprocessing`)绕开,进程间用队列或共享内存传数据。

        F = self._F(R, am, dt)
        self.P = F @ self.P @ F.T + self.Q * dt

### 一、三个传感器能否共用一个串口

**能，而且这正是推荐做法。** 你的物理连接已经是这样了——三个传感器都接在 STM32 上，STM32 和树莓派之间只有一条串口。所以本质上不是"三个传感器共用串口"，而是 **STM32 把三路数据汇聚后，通过这一条串口转发** 。

STM32 在这里做的是数据聚合器:它分别从各自接口读三个传感器(IMU/磁力计走 SPI/I2C,RTK 走另一个 UART),打上统一时间戳,再封装成帧从串口发给树莓派。

关键是 **怎么组帧** 。因为三个传感器速率差别很大(IMU 200Hz、磁力计几十 Hz、RTK 1–10Hz),不要把它们硬凑成一个大帧。更好的做法是 **按类型分帧,每帧带类型标识和时间戳**:

```
帧结构:  [0xAA 0x55] [type] [len] [timestamp_us(4B)] [payload...] [CRC16]
            帧头       类型   长度   STM32硬件时间戳        数据         校验

type=0x01: IMU    payload = ax,ay,az,gx,gy,gz (6×float = 24B)
type=0x02: MAG    payload = mx,my,mz (3×float = 12B)
type=0x03: RTK    payload = lat,lon,alt,vn,ve,vd,status (...)
```

这样 IMU 帧高频发、RTK 帧低频发、各走各的,树莓派按 type 分发到 ESKF 的不同入口。 **每帧独立带时间戳** 是后面时间同步的基础。

几个组帧要点:

- 帧头要能抗误同步,用两字节(0xAA 0x55)比单字节可靠,解析时找到帧头还要靠 len + CRC 二次确认。
- 一定要 **CRC 校验** 。串口在高波特率/长线/有干扰时会偶发误码,一个 bit 翻转喂进 ESKF 可能让位置跳一下。校验失败就丢帧。
- 用二进制定长 payload,别用 ASCII/JSON——后者体积大 2–3 倍,还要解析浮点字符串,白白吃带宽和 CPU。

### 二、波特率要求

先算 **实际需要多少带宽**,再选波特率。

以 IMU 200Hz 为主力估算单帧开销:帧头2 + 类型1 + 长度1 + 时间戳4 + payload24 + CRC2 = **34 字节/帧** 。

```
IMU:  200Hz × 34B = 6,800 B/s
MAG:   50Hz × 22B = 1,100 B/s
RTK:   10Hz × ~40B =  400 B/s
合计 ≈ 8,300 B/s
```

串口是 8N1,**每字节占 10 bit** (1起始 + 8数据 + 1停止),所以:

```
需要的比特率 ≈ 8,300 × 10 = 83,000 bps
```

看起来 115200 够用(占用率 ~72%)。 **但 72% 占用率太满了**,没有余量应对突发(比如 RTK 某帧变长、偶尔重传、调试时临时加数据),链路一满就开始丢帧或堆积延迟。经验法则是 **占用率控制在 50% 以下** 。

所以:

- **115200 勉强够,但不留余量,不推荐。**
- **460800 推荐** ——占用率降到 ~18%,余量充足,F405 的 USART 和树莓派都稳定支持。
- 如果以后 IMU 提到 500Hz 或加更多数据,直接上 **921600** 。

注意两点:

- F405 USART 的波特率由 APB 时钟分频得到,460800/921600 这些非标准率可能有小的分频误差,要确认误差在 ±2% 以内(一般没问题)。两端波特率必须严格一致。
- 树莓派的 GPIO 串口(`/dev/ttyAMA0`,PL011)比 mini-UART(`/dev/ttyS0`)稳定,**优先用 PL011** 。Zero 2W 默认 PL011 可能被蓝牙占用,需要在 `config.txt` 里 `dtoverlay=disable-bt` 把硬件串口让给 GPIO,并关掉串口控制台。

### 三、时间同步——树莓派端最容易出错、也最影响精度的部分

这是三个问题里最关键的。分几个层面:

**1\. 时间基准:全程用 STM32 的时间戳,不要用树莓派的接收时刻。**

前面强调过,这里再具体说为什么。树莓派收到帧的时刻 = 采样时刻 + SPI读取 + STM32组帧 + UART传输 + Linux内核缓冲 + 调度延迟。后面那几项是 **抖动的** (几 ms 不等),用接收时刻算 IMU 积分的 Δt,误差直接进状态。而 STM32 的硬件定时器时间戳是在 **采样那一刻** 打的,稳定到 μs。所以:

```python
dt = (pkt.stm32_ts - last_stm32_ts) * 1e-6   # 对的
# dt = time.monotonic() - last_recv_time      # 错的!别这么干
```

**2\. 时间戳在哪里打:越靠近采样越好。**

理想情况是 STM32 用 IMU 的\*\*数据就绪中断(DRDY)\*\*触发读取,在中断里第一时间锁存定时器值作为该样本的时间戳。如果是定时器轮询读 IMU,就用触发读取的那个定时器节拍做时间戳。别在"组完帧准备发送"时才取时间——那时已经晚了组帧的时间。

**3\. STM32 时间戳的来源:用一个不溢出/可处理溢出的自由运行定时器。**

开一个 32 位定时器,1MHz 计数(1μs 分辨率),32 位约 71 分钟溢出。如果作业时间长,要么用 64 位软件扩展(溢出时中断里累加高 32 位),要么在协议里处理回绕。树莓派端要按同样逻辑还原连续时间。

**4\. 三个传感器之间的时间对齐:它们的时间戳必须同源。**

IMU、磁力计、RTK 三帧的时间戳必须都来自 **STM32 的同一个定时器**,这样它们才在同一时间轴上,ESKF 才能正确地把不同时刻的观测对齐。这是用 STM32 统一打戳相对于"各传感器自带时间"的最大好处。

**5\. RTK 的时延补偿:观测要更新到它对应的历史时刻,不是当前时刻。**

RTK 解算本身有延迟(模块内部算 + 星基改正),它输出的位置其实是 **几十到上百 ms 之前** 那个历元的。如果你拿这个"旧"位置去和"当前"INS 状态算残差,就错位了。正确做法:RTK 帧里带的时间戳应该是 **该解对应的 GNSS 历元时刻** (STM32 收到 RTK 模块输出时,用模块给的历元或 PPS 对齐打戳),ESKF 更新时用那个时刻的 INS 状态做残差。

进阶做法是 **状态缓存/回溯**:ESKF 维护最近一两秒的状态历史,RTK 来了先回到它对应时刻做更新,再把之后的 IMU 重新积分到当前。这样最严谨,但实现复杂;一般场景下,只要 RTK 时延小且稳定,用对应历元时间戳做更新就够了。

**6\. PPS 秒脉冲:让 STM32 的时间轴和 GNSS 绝对时间挂钩。**

把 RTK 模块的 PPS(秒脉冲,每秒上升沿精确对齐 GNSS 整秒)接到 STM32 的定时器输入捕获。这样 STM32 的本地时间戳能校准到 GNSS 时间,既能修正 STM32 时钟的漂移,也让 RTK 历元和 IMU 时间戳精确对齐。对 cm 级系统,PPS 强烈建议接上。

**7\. 乱序与缓冲:树莓派解析时按时间戳排序处理。**

正常情况下帧是顺序到达的,但如果你用了多个传感器分帧 + 不同速率,偶尔可能出现处理顺序问题。融合主循环应该 **按时间戳单调推进**:IMU 帧驱动 predict,观测帧(RTK/MAG)按其时间戳插到对应位置更新。简单实现就是保证 predict 的 dt 永远来自单调递增的 IMU 时间戳,观测帧用自己的时间戳判断是否需要先补一步 predict。
